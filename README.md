# 🏆 Data Fusion Contest 2026 - Кейс 3 "Герои"

## 🦸‍♂️ Эволюция Решения: от Greedy к Google OR-Tools 🦸‍♀️

Данный репозиторий содержит две версии алгоритма для решения задачи маршрутизации героев (VRPTW). Проект прошел путь от интуитивного жадного поиска до строгой математической оптимизации.

---

## 📅 Версия 1: Жадный поиск (Baseline)

*Первая стабильная версия, основанная на быстрой эвристике.*

### 🎯 Основная стратегия

**Жадный алгоритм с поиском ближайшего соседа**, учитывающий:

* ⏰ **Временные окна** посещения объектов.
* 💰 **Стоимость найма** героев (отсечка по окупаемости).
* 🚶‍♂️ **Очки хода** и их перенос между днями.

### 🚀 Почему это было эффективно?

* **Соблюдение ограничений:** Учет `visit_cost` и жестких окон `day_open`.
* **Правило "Последнего шага":** Реализация логики посещения «на флажке» (когда очков меньше 100, но они есть).
* **Приоритет времени:** Сортировка по формуле `(t_day * 10000) + distance` — герои в первую очередь собирают то, что скоро исчезнет.
* **Скорость:** Мгновенная генерация решения за счет использования `NearestNeighbors` с манхэттенской метрикой.

---

## 💎 Версия 2: Математическая Оптимизация (Advanced)

*Текущая версия: Переход к использованию профессиональных солверов (Google OR-Tools).*

В этой версии мы отказались от простых эвристик в пользу глобального поиска оптимальных путей в пространстве и времени.

### 🎯 Что изменилось в стратегии?

**Математическая модель VRPTW:** Вместо выбора "ближайшего здесь и сейчас", алгоритм строит маршруты, минимизируя общие затраты на всей дистанции 7 дней.

### 🚀 Ключевые фишки новой версии:

1. ⚙️ **Google OR-Tools Core:** Использование стратегий `Guided Local Search` и `Parallel Cheapest Insertion` для нахождения глобального оптимума.
2. 🧠 **Двухэтапная оптимизация:**
* **День 1:** Маршрутизация со стартом из Замка.
* **Дни 2-7:** Динамическое построение матриц расстояний от точек завершения предыдущего дня.


3. 🔄 **Post-processing (Локальный поиск):** Кастомный модуль `LocalSearchOptimizer` после работы солвера пытается перераспределить объекты так, чтобы выкинуть самого дорогого (последнего) героя и сэкономить 2500 золота.
4. 📈 **Grid Search параметров:** Класс `ConfigTester` автоматически перебирает количество героев (18–21) и лимиты времени, выбирая лучший `score`.

## 📊 Модуль Глубокой Аналитики (`визуализация.ipynb`)

Для поддержки разработки был создан аналитический комплекс, генерирующий **14 дашбордов**:

* 🗺️ **Spatial Analysis:** Проекции `MDS` и `t-SNE` для визуализации карты мельниц.
* 🔬 **Clustering:** Кластеризация `K-Means` для выделения зон ответственности героев.
* 🕸️ **Graph Networks:** Сетевой анализ плотности объектов через `NetworkX`.
* 💰 **Efficiency Waterfall:** Графики окупаемости каждого нанятого героя.

## 📁 Структура файлов

* `code_ortools.py` — **Основной скрипт (V2)**. Полный цикл: загрузка -> оптимизация -> локальный поиск -> сохранение.
* `визуализация.ipynb` — **EDA-модуль**. Инструмент для поиска инсайтов и генерации графиков.
* `submission.csv` — Результат работы алгоритма.
